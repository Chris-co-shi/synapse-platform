# Synapse Platform 架构原则

本文是 Synapse Platform v1 的架构原则基线。具体架构决策由 ADR 记录；任何 ADR 都不得无说明地违反本文原则。

## P-001 平台不承载业务领域

Synapse Platform 是企业数字化基座，不是 MES、WMS、QMS、EMS、MOM、ERP 或 IoT 业务系统。

Platform 可以提供业务系统需要的公共能力，但不能把具体业务流程、业务规则、业务实体或业务数据库纳入平台核心。

## P-002 业务系统自治，平台能力复用

业务系统拥有自己的领域模型、数据库、业务流程、发布节奏和运行边界。

Platform 提供统一身份、权限、入口、日志、审计、监控、任务、文件、消息和集成等公共能力。业务系统通过 API、Client、事件、Webhook 或连接器接入 Platform，而不是直接共享 Platform 数据库。

## P-003 Framework 与 Platform 解耦

Synapse Framework 是技术底座，Synapse Platform 是可运行的平台产品。

Framework 不允许包含 IAM 业务、Gateway 服务、Platform 数据库业务模型或 Platform 业务代码。Platform 可以依赖 Framework；Framework 不能反向依赖 Platform。

## P-004 控制面优先于功能堆叠

Synapse Platform 的核心价值是统一企业数字化基础设施，而不是功能数量。

优先保证身份、安全、入口、权限、日志、审计和可运维闭环，再扩展消息、文件、任务、集成、报表、工作流或 AI Agent 等能力。

## P-005 API First / Contract First

跨服务、跨系统和跨团队的交互必须先稳定契约，再实现内部代码。

`*-api` 保存稳定契约，`*-client` 保存调用适配，`*-server` 承载实现。禁止通过共享 Entity、Mapper、Repository 或数据库替代契约。

## P-006 默认安全，显式放行

平台安全默认采用 fail closed。

除明确白名单外，入口默认需要认证；下游服务必须独立验证令牌并执行资源授权。Gateway 完成认证不能成为下游服务跳过鉴权的理由。

## P-007 Gateway 是统一入口，不是业务授权中心

Gateway 负责统一入口、路由、基础认证校验、Header 清理、可信入口证明和基础流量治理。

Gateway 不查询业务数据库，不执行业务资源权限，不承担组织、租户、数据范围或资源归属判断。

## P-008 IAM 是身份与授权源，不是所有权限执行点

IAM 负责主体、凭据、Client、Token、角色、权限和授权信息。

接口权限、资源权限、数据权限和资源归属判断由资源所在服务兜底执行。

## P-009 可部署性是一等架构属性

平台在设计阶段必须考虑本地开发、企业私有化和云原生演进。

具体部署方式是否进入 V1 正式支持矩阵，应由部署文档确认；未经验证的部署方式不能描述为已交付能力。

## P-010 可观测性是一等架构属性

关键平台能力必须支持 traceId、操作主体、请求入口、服务调用、异步任务和审计日志的关联。

日志、审计、指标、健康检查和运行状态不是后补功能，而是平台可运维性的基础。

## P-011 私有化优先，避免厂商绑定

Synapse Platform 必须优先满足企业本地化和私有化部署需求。

平台设计不能绑定特定服务器、固定路径、单一中间件实例或特定云厂商，并应保留云原生演进能力。

## P-012 简单优先，但不牺牲正确性

产品和架构应降低企业数字化复杂度。

不为了形式引入不必要的服务、组件、抽象或流程；同时不能用“简单”作为跳过安全、审计、测试、部署和边界设计的理由。

## P-013 先确认边界，再扩大能力

任何新增模块或重要能力进入设计前，都必须回答：

1. 它解决什么企业问题；
2. 它是否属于多个业务系统可复用的平台能力；
3. 它是否会让 Platform 承载具体业务领域；
4. 它是否进入 V1，还是仅属于路线图候选。

未回答清楚前，不进入正式开发。

## Change Rule

在架构冻结前，原则、边界和模块可以通过评审调整。

架构冻结后，任何影响产品定位、服务边界、安全模型、部署模型、网络边界或核心数据流的修改，都必须通过 ADR 记录。
