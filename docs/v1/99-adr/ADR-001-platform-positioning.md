# ADR-001：Synapse Platform 产品定位与边界

## Status

Accepted

## Context

Synapse Platform 的目标不是构建一个 Spring Cloud Demo，也不是把 MES、WMS、QMS、EMS、MOM、ERP 周边或 IoT 等业务系统合并成一个大一统应用。

企业在数字化建设初期，或者已经拥有多套彼此割裂的系统时，通常会出现身份、权限、入口、日志、监控、任务和系统接入能力重复建设的问题。平台需要先明确自己的产品定位和边界，避免后续因为功能增加而逐步演变为业务大杂烩。

产品定义的权威来源为 [`00-product/product.md`](../00-product/product.md)。本 ADR 记录为什么选择该定位。

## Why

如果不先明确平台定位，后续会持续产生以下争议：

- Platform 是否应该直接承载 MES、WMS 或其他业务模块；
- IAM 是否应该替所有业务系统执行业务授权；
- Gateway 是否应该读取业务数据并判断资源归属；
- Framework 与 Platform 是否可以互相污染；
- Task、Message、File、Audit、Monitor 和 Integration 等能力是否属于平台；
- 已有企业系统应被替换，还是以标准方式逐步接入和整合。

ADR-001 用于冻结最高层产品边界，为后续架构、模块设计、测试和交付提供判断依据。

## Alternatives

### 方案 A：统一业务大平台

把 MES、WMS、QMS、EMS、MOM、ERP 周边和 IoT 等业务能力全部放入 Synapse Platform。

优点：

- 初期功能看起来完整；
- 演示和单项目集成较直接。

缺点：

- 平台范围持续膨胀；
- 业务系统失去自治；
- 数据库、领域模型和发布节奏高度耦合；
- 任意业务变化都会影响平台；
- 难以作为可复用的企业基座长期演进。

结论：不采用。

### 方案 B：微服务脚手架或依赖集合

把 Synapse Platform 定义为 Spring Cloud 工程模板，只提供网关、认证、依赖管理和样例服务。

优点：

- 实现成本低；
- 容易快速启动项目。

缺点：

- 不具备完整平台产品价值；
- 不能解决企业多系统统一治理问题；
- 难以形成可部署、可运维、可交付的产品。

结论：不采用。

### 方案 C：企业数字化平台

将 Synapse Platform 定位为完整的平台产品，为企业提供统一身份、安全、入口、治理和共享平台能力，帮助企业建设统一的数字化基础设施，并逐步收口和整合分散系统。

优点：

- 产品价值和边界清晰；
- 业务系统可以保持自治；
- 公共能力可以复用；
- 支持企业本地化与私有化部署；
- 能从开源项目逐步演进为可商业交付产品。

代价：

- 前期产品和架构设计成本更高；
- 需要持续维护文档、ADR、测试和交付规范；
- 必须克制无边界扩张。

结论：采用。

## Decision

Synapse Platform 定位为：

> 一个企业数字化平台（Enterprise Digital Platform），为企业 IT 提供统一身份、安全、入口、治理与共享平台能力，帮助企业构建统一的数字化基础设施，并对分散的业务系统进行统一接入、统一治理和逐步整合。

产品 Slogan：

> **Build Your Enterprise Digital Foundation.**

产品核心价值包括：

- Unified：统一企业数字化基础设施；
- Open：开源、开放、可扩展；
- Self-hosted：支持企业自主和私有化部署；
- Simple：直观、易操作、易理解；
- Evolvable：支持企业持续演进。

Synapse Platform 是完整的平台产品，不是业务系统、开发框架或微服务脚手架。

平台优先服务制造业数字化场景，但产品定义不绑定单一业务领域。

## Product Boundary

Platform 可以提供：

- 统一身份与访问管理；
- OAuth 2.0 / OIDC；
- RBAC；
- 统一入口；
- 日志、审计、监控与可观测性；
- 分布式任务与调度；
- 文件、消息、配置和集成等共享能力；
- 面向业务系统的 API、Client、事件和连接器；
- 安装、部署、运维和交付基线。

具体能力是否进入 V1，必须由产品范围和模块设计单独确认。

Platform 不负责：

- MES、WMS、QMS、EMS、MOM 或 ERP 核心业务；
- 业务系统领域模型、数据库、业务规则和业务流程；
- 用共享数据库替代系统集成；
- 以低代码、完整 BI、完整 BPM 或 AI Agent 平台作为当前核心产品定义。

AI Agent 可以作为未来平台能力候选，但当前不自动构成 V1 交付承诺。

## Impact

该决策影响：

- 产品文档和 GitHub README；
- Platform 模块及服务边界；
- IAM、Gateway、Audit、Monitor、Task、File、Message 和 Integration 的职责；
- 业务系统接入方式；
- 部署、测试和交付文档；
- 后续 ADR 和 Architecture Freeze 标准。

## Consequences

正向影响：

- 平台产品定位稳定；
- 企业价值不再由技术组件数量定义；
- 业务系统保持自治；
- 文档、开发、测试和交付拥有统一上位依据。

约束：

- 不能为了快速展示而把业务能力塞入 Platform；
- 每个新增模块都必须证明其平台公共价值；
- 未确认的功能、部署方式和商业策略不能写成交付事实。

## Follow-ups

后续需要逐步确认并记录：

- V1 产品范围；
- Platform 总体架构和服务边界；
- Gateway、IAM 与权限执行边界；
- 日志、审计、监控和可观测性边界；
- 部署支持矩阵；
- 业务系统和外部系统接入模型；
- 测试、验收和交付基线。
