# Synapse Platform V1 范围

## Document Metadata

| Item | Value |
| --- | --- |
| Audience | 产品负责人、架构师、开发、测试、实施、运维和交付人员 |
| Purpose | 明确 Synapse Platform V1 的交付目标、优先级、范围边界和完成标准 |
| Scope | Synapse Platform 第一个可运行开源版本 |
| Status | Accepted |

本文档是 Synapse Platform V1 范围的单一事实来源。任何模块、能力或交付项进入 V1 前，都必须符合本文档定义。

## 1. V1 Product Goal

Synapse Platform V1 的目标是交付一个：

> **可运行、可测试、文档完整、能够通过 Docker Compose 安装和启动的开源版本。**

V1 首先完成平台自身闭环。外部 MES、WMS、QMS、EMS、MOM、ERP 周边、IoT 或第三方业务系统接入，不作为 V1 完成条件。

只有平台自身能够稳定运行并完成核心流程后，才进入真实业务系统接入验证阶段。

## 2. Priority Model

V1 使用以下优先级：

| Priority | Meaning |
| --- | --- |
| `P0` | V1 核心闭环和发布阻断项；未完成则 V1 不成立 |
| `P1` | V1 正式范围中的低优先级能力；应在 V1.0 发布前完成，但不得阻塞 P0 闭环建设 |
| `P2` | 仅保留基础能力或后续扩展方向；不作为 V1 正式交付承诺 |
| `Out of Scope` | 明确不进入 V1 |

## 3. P0 — Core Release Scope

### 3.1 Gateway

Gateway 是 V1 必须交付的统一入口。

V1 至少需要形成：

- 统一访问入口；
- 基础路由能力；
- 认证链路接入；
- 不可信 Header 清理；
- 受保护资源访问控制入口；
- 与 IAM 和资源服务形成完整访问链路；
- 必要的入口日志和可追踪信息。

Gateway 不承担具体业务资源授权，也不读取业务数据库完成权限判断。

### 3.2 IAM

IAM 是 V1 必须交付的身份与访问管理核心服务。

V1 至少需要形成：

- 平台管理员初始化；
- 用户管理；
- 角色管理；
- 权限管理；
- 用户、角色和权限关系管理；
- 登录；
- Access Token 签发；
- Refresh Token 刷新；
- 退出；
- 当前用户信息；
- OAuth 2.0 / OpenID Connect 基础闭环。

具体协议流、Token 策略和 Client 模型由后续安全架构和 IAM 设计文档确认。

### 3.3 RBAC and Authorization

V1 必须形成完整的 RBAC 管理与执行闭环：

- 用户、角色、权限的管理关系可配置；
- 用户能够通过角色获得权限；
- 受保护接口能够执行权限判断；
- 有权限请求成功；
- 无权限请求被拒绝；
- Gateway 完成入口校验后，下游资源服务仍执行独立令牌验证和权限兜底。

V1 不要求在 Platform 中集中执行所有业务系统的数据权限和资源归属判断。

### 3.4 Resource Server Security

V1 平台内部受保护服务必须按照 Resource Server 模型完成安全闭环：

- 独立验证 Access Token；
- 拒绝无效或失效 Token；
- 执行服务端权限校验；
- 不依赖前端隐藏按钮作为安全措施；
- 不因为请求经过 Gateway 就跳过鉴权。

### 3.5 Logging and Audit

V1 必须包含以下三类平台记录：

| Type | Purpose |
| --- | --- |
| 操作日志 | 记录用户在平台管理端和平台服务中的关键操作 |
| 安全日志 | 记录登录成功、登录失败、刷新、退出、拒绝访问等安全事件 |
| 审计日志 | 记录可追溯的关键主体、行为、对象、结果和上下文 |

V1 需要能够通过管理端查看和检索上述记录。

运行日志、指标、Trace 等可观测性基础能力应支撑问题定位，但其完整范围由后续可观测性文档确认。

### 3.6 Management Console

管理端前端属于 V1 必须交付范围。

管理端至少应支持完成 V1 核心闭环所需的操作：

- 管理员登录；
- 用户管理；
- 角色管理；
- 权限管理；
- 用户角色分配；
- 角色权限分配；
- 查看当前用户信息；
- 查看操作日志；
- 查看安全日志；
- 查看审计日志；
- 查看基础平台运行状态。

管理端的详细信息架构、页面范围和交互标准由后续产品与设计文档确认。

### 3.7 Docker Compose Deployment

Docker Compose 是 V1 正式支持的部署方式。

V1 必须提供：

- 可执行的 Docker Compose 部署编排；
- 明确的环境变量和配置说明；
- 依赖中间件启动说明；
- 数据初始化方式；
- 安装和启动步骤；
- 健康检查或基本运行验证方式；
- 停止、重启和清理说明；
- 常见故障排查说明。

VM、Kubernetes 和其他部署方式不作为 V1 正式交付阻断项。

## 4. P1 — Lower-priority V1 Scope

以下能力属于 V1 正式范围，但优先级低于 Gateway、IAM、RBAC、日志审计、管理端和 Docker Compose。

### 4.1 File

V1 目标是形成可运行的基础文件服务闭环。具体上传、下载、元数据、存储适配、安全和配额范围由 File 模块设计确认。

### 4.2 Message

V1 目标是形成可运行的基础消息能力。具体站内消息、模板、发送渠道、状态和重试范围由 Message 模块设计确认。

### 4.3 Task

V1 目标是形成可运行的基础任务能力。具体定时任务、分布式调度、执行日志、失败重试和管理范围由 Task 模块设计确认。

File、Message、Task 在 P0 核心闭环稳定之后进入主要开发阶段，但应在 V1.0 正式发布前完成其经确认的最小功能范围。

## 5. P2 — Foundation Only

### 5.1 Monitor

Monitor 在 V1 中只要求基础运行可见性，不建设完整监控产品。

V1 可以包含：

- 服务健康状态；
- 基础运行指标；
- 服务或组件可用状态；
- 管理端基础状态查看入口。

告警中心、复杂指标分析、完整 APM、长期时序数据管理等不自动进入 V1。

### 5.2 Config

V1 仅保留平台自身正常运行所需的配置能力。

不在 V1 中建设独立、完整的企业配置中心产品，也不将所有业务系统配置统一纳入 Platform 管理。

## 6. Out of Scope for V1

以下能力不进入 V1 正式范围：

- 外部业务系统真实接入验证；
- Integration 完整平台能力；
- Workflow；
- MDM；
- Report；
- 低代码平台；
- 完整 BPM 产品；
- 完整 BI 产品；
- AI Agent 平台；
- 复杂多租户；
- 多园区统一治理；
- 多 Region；
- 完整高可用和灾备体系；
- Service Mesh；
- Kubernetes 正式交付；
- 商业版和企业版能力划分。

这些能力可以进入后续路线图，但不得影响 V1 核心闭环和交付节奏。

## 7. V1 Self-contained Acceptance Flow

V1 至少应能够完成以下平台自身闭环：

```text
通过 Docker Compose 启动完整平台
  ↓
初始化平台管理员
  ↓
管理员登录管理端
  ↓
创建用户、角色和权限
  ↓
为用户分配角色，为角色分配权限
  ↓
用户登录并获得 Access Token / Refresh Token
  ↓
通过 Gateway 访问受保护的平台资源
  ↓
有权限请求成功
  ↓
无权限请求被拒绝
  ↓
刷新 Token
  ↓
退出后相关会话或令牌按既定策略失效
  ↓
管理端查看操作日志、安全日志和审计日志
  ↓
查看平台基础运行状态
```

File、Message、Task 需要分别具备经其模块设计确认的最小自闭环验收场景。

## 8. Definition of Done

Synapse Platform V1 只有在以下条件全部满足后，才视为完成：

### 8.1 Functionality

- P0 能力全部可用；
- P1 能力完成其确认后的最小范围；
- 平台自身验收链路可以完整跑通；
- 不依赖未交付的外部业务系统才能运行。

### 8.2 Testing

- 核心单元测试通过；
- 核心集成测试通过；
- 身份、权限、Token 和访问拒绝场景测试通过；
- 操作日志、安全日志和审计日志测试通过；
- Docker Compose 安装与启动验证通过；
- 测试范围和结果可追溯。

具体覆盖率目标和测试分层由测试策略文档确认。

### 8.3 Documentation

至少应具备：

- 产品定义和 V1 范围；
- 总体架构和关键 ADR；
- 开发与接口规范；
- 核心模块设计；
- 测试策略和验收说明；
- Docker Compose 部署文档；
- 安装、初始化、运维和故障排查说明；
- 版本说明。

### 8.4 Installation and Operation

- 新环境可以依据文档完成安装；
- Docker Compose 可以启动 V1 必需服务和依赖；
- 初始化流程可重复执行或明确保护；
- 平台能够完成基础健康验证；
- 常见启动和配置问题有排查依据。

## 9. Scope Change Rule

新增能力进入 V1 前，必须回答：

1. 是否是平台自身闭环不可缺少的能力；
2. 是否阻塞可运行开源版本发布；
3. 是否已经明确最小交付范围；
4. 是否具备测试和文档验收方式；
5. 是否会导致 V1 范围失控。

影响 P0、P1、V1 完成标准或明确不做事项的变更，必须更新本文档，并在必要时新增 ADR。
