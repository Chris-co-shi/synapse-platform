# Synapse Platform V1.0 功能清单与优先级

文档角色：产品经理  
版本：V1.0 二阶段 Framework 对齐版  
适用范围：Synapse Platform 企业级基座平台  
参考基础：Synapse Framework 二阶段已完成技术基座  
目标：明确 V1.0 做什么、先做什么、暂缓什么、如何验收

---

## 1. 产品定位

Synapse Platform V1.0 定位为：

> 面向企业多业务系统的统一能力基座平台。

它不是 MES、WMS、QMS、EMS、CRM、ERP 等业务系统本身，而是为这些业务系统提供统一的：

- 网关入口；
- 身份认证；
- 用户、组织、角色、权限；
- 系统、菜单、按钮、API 资源；
- 操作上下文；
- 审计追踪；
- 配置字典；
- 消息、文件、任务等通用能力；
- SDK / Client / 接入规范。

V1.0 的核心目标不是“大而全”，而是先完成企业平台的第一条可用闭环：

```text
登录认证
  -> Gateway 统一入口
  -> 可信上下文注入
  -> 服务端权限兜底
  -> 操作审计
  -> 业务系统按 SDK 接入
```

---

## 2. 与二阶段 Framework 的关系

Synapse Platform 必须基于二阶段 Framework，而不是继续沿用一阶段模块口径。

二阶段 Framework 已提供以下技术支撑：

| Framework 模块 | Platform 使用方式 |
|---|---|
| `synapse-bom` | Platform BOM import 的基础版本源 |
| `synapse-core` | ErrorCode、Exception、Result、OperationContext、ID 等核心契约 |
| `synapse-webmvc` | IAM、Audit、Config、Message、File、Task 等 Servlet MVC 服务的 Web 支撑 |
| `synapse-webflux` | Gateway 的 WebFlux Trace、异常响应、Reactor Context 支撑 |
| Spring Cloud OpenFeign | 由 Platform 直接使用官方组件；Framework 已删除原 `synapse-cloud` |
| `synapse-time` | UTC 时间、业务日期、时区、查询范围转换 |
| `synapse-config` | 配置读取抽象，不是配置中心 |
| `synapse-i18n` | 国际化消息解析抽象，不是国际化资源中心 |
| `synapse-data` | 数据层自动填充、审计字段支撑 |
| `synapse-cache` | Redis、缓存、锁、限流、幂等基础设施 |
| `synapse-security` | trusted-header、AuthenticatedUser、PermissionChecker、权限注解适配 |
| `synapse-oauth2` | JWT / JWK / OAuth2 技术能力 |
| `synapse-audit` | 审计事件契约 |
| `synapse-messaging` | Broker 中立的消息契约与发布消费编排 |
| Platform File | 文件能力已归 Platform，Framework 不再提供 `synapse-file` |

Platform 负责把这些技术能力组合成可启动平台服务。

## 3. V1.0 产品目标

V1.0 要解决的问题：

1. 新业务系统不再重复开发登录认证；
2. 新业务系统不再重复开发用户、组织、角色、权限；
3. 所有前端和外部请求统一通过 Gateway；
4. Gateway 能校验 Token 并注入可信 OperationContext；
5. 下游服务能恢复当前用户、TraceId、actor、initiator；
6. 后端接口能通过权限注解或拦截器做兜底校验；
7. 用户登录后能看到自己有权限访问的系统、菜单、按钮；
8. 关键操作可审计、可追踪、可按 TraceId 检索；
9. 业务系统通过 `*-client` 接入平台能力，而不是直接访问平台数据库；
10. Platform 服务只依赖 Framework，Framework 不反向依赖 Platform；
11. 时间字段、业务日期、时区查询统一遵守 Framework 二阶段时间边界；
12. Message / File / Task / Report 等能力在边界清晰后逐步平台化。

## 4. 优先级定义

| 优先级 | 含义 | 判断标准 |
|---|---|---|
| P0 | V1.0 必须完成 | 没有它平台无法形成登录、鉴权、上下文、审计闭环 |
| P1 | V1.0 应完成 | 不阻塞主闭环，但影响企业内部可用性 |
| P2 | V1.0 可选 / V1.1 候选 | 有价值，但容易扩大范围，可在主闭环稳定后做 |
| P3 | 后续规划 | 不进入 V1.0 主线 |
| 暂不做 | 明确排除 | 做了会导致平台范围失控 |

## 5. V1.0 成功闭环

```text
管理员初始化平台
  ↓
创建系统、菜单、按钮、API 资源
  ↓
创建用户、组织、角色
  ↓
给角色授权，给用户分配角色
  ↓
用户登录 IAM
  ↓
IAM 签发 access token / refresh token
  ↓
前端访问 Gateway
  ↓
Gateway 校验 Token，清洗伪造 Header
  ↓
Gateway 注入可信 OperationContext Header
  ↓
Platform / Business 服务恢复 OperationContext
  ↓
服务端执行权限校验
  ↓
执行业务操作
  ↓
Data 自动填充 createdBy / updatedBy
  ↓
Audit 记录登录、操作、权限变更日志
  ↓
按 TraceId 串联请求、服务调用、异步任务、审计日志
```

# 7. P0 功能清单

## 7.1 Platform 工程基线

### 功能定位

为后续平台服务建立统一工程、依赖、文档和边界约束。

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 父工程 `pom.xml` | P0 | 多模块 reactor 管理 |
| 根工程 `dependencyManagement` | P0 | import `synapse-bom`，统一管理 Platform 自身依赖版本 |
| `AGENTS.md` | P0 | 约束 Codex / AI 编码助手 |
| docs 目录 | P0 | 产品、架构、任务、API、部署文档 |
| 模块命名规范 | P0 | 服务按 `synapse-{capability}-platform` 与 `*-api` / `*-client` / `*-server` 拆分 |
| 时间规范 | P0 | Instant / LocalDate / ZoneId / UTC 半开区间 |
| 错误码规范 | P0 | Framework / Platform / Business 分层归属 |

### 验收标准

1. Platform 可以独立 clone、build、validate；
2. 根工程可以 import Framework BOM；
3. Gateway 不会因为依赖错误引入 MVC；
4. WebMVC 服务统一依赖 `synapse-webmvc`；
5. 不出现统一大杂烩 `synapse-platform-api`；
6. 不创建 `synapse-platform-common` 和 `synapse-platform-bom`。

---

## 7.2 Gateway 网关中心

### 功能定位

Gateway 是所有前端请求、外部请求进入 Platform 和业务系统的统一入口。

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 统一路由转发 | P0 | 路由到 IAM、Resource、业务服务 |
| Token 校验 | P0 | 校验 access token |
| 匿名接口白名单 | P0 | 登录、健康检查等接口放行 |
| 请求 Header 清洗 | P0 | 清除外部伪造身份 Header |
| OperationContext Header 注入 | P0 | 注入 actor、initiator、traceId、source 等 |
| TraceId 生成 / 透传 | P0 | 每个请求必须有 TraceId |
| 统一网关错误响应 | P0 | 使用 `synapse-webflux` 的响应边界 |
| CORS | P0 | 支持 Admin Console 调用 |
| 基础限流 | P1 | 使用 `synapse-cache` 能力 |
| 黑白名单 | P1 | IP / path / system 维度 |
| 访问日志 | P1 | 请求路径、用户、耗时、状态 |
| 动态路由 | P2 | 后台配置路由后动态生效 |

### 验收标准

1. Gateway 是 WebFlux / Spring Cloud Gateway 服务；
2. Gateway 允许依赖 `synapse-webflux`；
3. Gateway 禁止依赖 `synapse-webmvc` 和 `synapse-data`；
4. 未登录访问受保护接口返回未认证；
5. Token 无效或过期返回统一错误；
6. 外部传入的身份 Header 不可信，必须清洗；
7. Gateway 校验通过后向下游注入可信 Header；
8. 错误响应包含 traceId；
9. Gateway 不做复杂 API 权限和数据权限决策。

---

## 7.3 IAM 身份权限中心

### 功能定位

IAM 是平台最核心服务，负责身份、会话、RBAC 权限和数据权限基础规则。

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 用户名密码登录 | P0 | 基础登录方式 |
| access token 签发 | P0 | 短期访问令牌 |
| refresh token 签发 | P0 | 刷新令牌 |
| Token 刷新 / 注销 | P0 | 刷新与失效 |
| 用户管理 | P0 | 新增、编辑、启停、重置密码 |
| 组织管理 | P0 | 公司、工厂、部门树 |
| 角色管理 | P0 | 角色、状态、授权 |
| 用户角色分配 | P0 | 多角色权限取并集 |
| 权限计算 | P0 | 用户权限码、菜单、按钮、API |
| 当前用户接口 | P0 | `/iam/users/me` |
| 当前权限接口 | P0 | 当前用户权限码集合 |
| 登录日志事件 | P0 | 成功、失败都要记录 |
| 在线会话 | P1 | 查询、强制下线 |
| 登录失败次数限制 | P1 | 防暴力破解 |
| 账号锁定 | P1 | 多次失败锁定 |
| LDAP / AD | P3 | 后续企业集成 |
| 钉钉 / 企微登录 | P3 | 后续扩展 |

### 验收标准

1. 用户名唯一；
2. 禁用用户不可登录；
3. 密码错误返回统一错误；
4. 登录成功签发 access token 和 refresh token；
5. 退出后 token 失效；
6. refresh token 失效后不可继续刷新；
7. 权限变更后缓存失效；
8. IAM 不处理文件存储、消息发送、任务调度等业务。

## 7.4 Resource 资源中心

### 功能定位

Resource 管理“有哪些资源可被访问”，IAM 管理“谁能访问”。

V1.0 建议：

```text
逻辑边界独立
物理部署先合入 synapse-iam-server
后续再拆 synapse-resource-server
```

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 系统管理 | P0 | MES、WMS、QMS 等系统注册 |
| 系统入口配置 | P0 | 前端入口地址、图标、状态 |
| 菜单管理 | P0 | 目录、页面、外链、路由 |
| 按钮权限管理 | P0 | 页面操作权限 |
| API 资源管理 | P0 | 接口路径、方法、权限码 |
| 角色系统授权 | P0 | 控制系统入口 |
| 角色菜单授权 | P0 | 控制菜单可见性 |
| 角色按钮授权 | P0 | 控制按钮显示 |
| 角色 API 授权 | P0 | 控制后端接口权限 |
| 资源同步日志 | P1 | 注解扫描 / 初始化导入预留 |
| 注解扫描注册 | P2 | 后续自动化注册 |

### 验收标准

1. 系统编码全局唯一；
2. 菜单、按钮、API 资源均有稳定权限码；
3. 用户登录后只返回授权系统和菜单；
4. 前端按钮权限只控制展示；
5. 后端 API 仍必须兜底校验；
6. Resource 表前缀使用 `res_`，即使物理合入 IAM。

## 7.5 OperationContext 当前操作上下文

### 功能定位

OperationContext 是统一追溯能力的核心，贯穿 HTTP、Feign、MQ、Task、Data、Audit。

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| HTTP 请求上下文恢复 | P0 | 从 Gateway Header 恢复 |
| 当前用户获取 | P0 | 服务可获取 actor |
| TraceId 获取 | P0 | 日志、异常、响应统一使用 |
| actor / initiator 区分 | P0 | 实际执行者 / 原始发起人 |
| Feign 上下文传播 | P0 | Platform 直接使用 Spring Cloud OpenFeign |
| 数据自动填充 | P0 | 使用 `synapse-data` 获取当前操作人 |
| MQ 上下文传播 | P1 | 使用 `synapse-messaging` 契约 |
| Task 上下文快照 | P1 | 异步任务保留 initiator |

### 验收标准

1. HTTP 同步请求可以获取当前用户；
2. 服务间 Feign 调用可以传播 TraceId 和 OperationContext；
3. data 自动填充可以拿到当前操作人；
4. 异步任务不会全部记录成 `system`；
5. ThreadLocal / Reactor Context 使用后必须清理；
6. 审计日志必须记录 actor 和 initiator。

## 7.6 Audit 审计中心 MVP

### 功能定位

审计中心记录平台关键行为，满足安全、合规、排障和追溯需求。

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 登录日志 | P0 | 登录成功、失败 |
| 操作日志 | P0 | 新增、修改、删除、导入、导出 |
| 权限变更日志 | P0 | 用户授权、角色授权、资源授权 |
| 审计事件接收 API | P0 | HTTP 方式先落地 |
| TraceId 查询 | P0 | 按 TraceId 串联日志 |
| 审计参数脱敏 | P0 | 密码、token、secret 必须脱敏 |
| MQ 事件消费 | P1 | 使用 `synapse-messaging` 后续增强 |
| 文件访问日志 | P1 | 上传、下载、删除 |
| 任务执行审计 | P1 | 触发、重试、取消 |
| 数据变更日志 | P2 | 修改前后对比 |

### 验收标准

1. 登录成功和失败都有日志；
2. 用户、角色、权限变更有日志；
3. 审计记录包含 actor、initiator、systemCode、traceId、occurredAt；
4. 审计时间使用 `Instant`；
5. 查询日期必须转换为 UTC 半开区间；
6. 普通用户不可删除审计日志。

## 7.7 Config / Dict 配置字典中心

### 功能定位

Config / Dict 管理平台和业务系统的业务配置、系统参数、字典、枚举、开关。

注意：

```text
Nacos
  = 技术配置、服务发现、Spring 配置

synapse-config
  = Framework 配置读取抽象，不是配置中心

synapse-config-service
  = Platform 业务配置中心
```

### 功能清单

| 功能 | 优先级 | 说明 |
|---|---:|---|
| 系统参数列表 | P0 | 查询参数 |
| 新增 / 编辑参数 | P0 | 创建、修改参数 |
| 字典类型管理 | P0 | 管理字典分类 |
| 字典项管理 | P0 | 管理枚举值 |
| 参数修改审计 | P0 | 修改必须记录 |
| 字典缓存 | P1 | 提升读取性能 |
| 敏感参数加密 | P1 | 密钥、密码等 |
| 参数刷新事件 | P2 | 变更后通知服务 |

### 验收标准

1. 参数 Key 唯一；
2. 字典类型编码唯一；
3. 同一字典类型下字典值唯一；
4. 停用字典项不影响历史数据展示；
5. 参数修改记录操作日志。

| 全部数据 | P1 | 管理员或特殊角色 |

## 8.2 Message 消息中心 MVP

# 10. V1.0 明确不做

| 内容 | 原因 |
|---|---|
| 完整低代码平台 | 范围过大，容易拖垮主线 |
| 完整 BPMN 引擎 | V1.0 不做复杂流程编排 |
| 完整 BI / 数据仓库 | Report MVP 不等于 BI 平台 |
| 插件市场 | 核心能力稳定前没有意义 |
| 完整 MDM | V1.0 只做组织、用户、字典等基础主数据 |
| 完整 SaaS 多租户 | 第一阶段优先私有化企业平台 |
| AI 平台 | 先不要为了 AI 而 AI |
| 复杂 API 网关商业化管理 | Gateway 先满足内部统一入口 |
| 复杂文档管理系统 | File Center 只做文件中心 MVP |
| OCR / 视频转码 | 不属于平台 V1.0 主线 |

# 12. V1.0 验收场景

## 12.1 基础平台验收

1. 管理员登录平台；
2. 创建组织；
3. 创建用户；
4. 创建角色；
5. 创建系统；
6. 创建菜单；
7. 创建按钮；
8. 创建 API 资源；
9. 给角色授权；
10. 给用户分配角色；
11. 用户登录后看到授权菜单；
12. 用户访问授权接口成功；
13. 用户访问未授权接口失败；
14. 所有关键操作有审计日志。

## 12.2 业务系统接入验收

以 `demo-business-service` 为例：

1. 注册 Demo 系统；
2. 配置 Demo 路由；
3. 配置 Demo 菜单；
4. 配置 Demo API 权限；
5. 创建 Demo 管理员角色；
6. 给用户分配 Demo 管理员角色；
7. 用户通过 Gateway 访问 Demo 接口；
8. Demo 服务获取当前用户；
9. Demo 服务写入审计字段；
10. Demo 服务调用 Message / File / Task Client；
11. 管理后台可以看到消息记录、文件记录、任务记录、操作日志。

## 12.3 异步追溯验收

1. 用户发起异步导出；
2. 系统创建任务；
3. 任务保存 OperationContext 快照；
4. 任务执行时 actor = TASK / SYSTEM；
5. 任务 initiator = 原始用户；
6. 任务完成后通知原始用户；
7. 审计日志可以看到真实发起人；
8. 通过 TraceId 可以串起请求、任务、消息。

# 13. 最终结论

Synapse Platform V1.0 的核心不是功能数量，而是建立企业基座平台的第一条完整链路：

```text
统一入口
  + 统一身份
  + 统一资源
  + 统一权限
  + 统一上下文
  + 统一时间边界
  + 统一审计
  + 统一接入规范
```

二阶段 Framework 已经把技术支撑边界拆清楚，Platform V1.0 的重点是把这些技术能力组合成可运行、可接入、可追溯的平台服务。
